## 1. 代码组织模式

### 关注点
- 目录结构反映了怎样的架构分层？（表现层、业务层、数据层？）
- 模块边界是如何划定的？（按功能？按技术关注点？按领域？）
- 是否有明确的依赖方向规则？（如外层依赖内层，内层不依赖外层）
- 公共导出与内部实现的隔离方式

### 常见模式
- **按功能分组**（feature-based）：每个功能目录包含自己的 UI、逻辑、数据
- **按层分组**（layered）：src/controllers、src/services、src/models
- **六边形架构**（hexagonal）：核心业务在中心，适配器在外围
- **模块化单体**：每个模块像独立服务，通过明确接口通信

### 识别信号
- `index.ts` / `__init__.py` 文件 → 模块公共接口
- `internal/` / `private/` 目录 → 封装边界
- `shared/` / `common/` 目录 → 共享基础设施
- barrel files（聚合导出文件） → 接口控制

### 输出要求
描述目录组织规则、模块间依赖方向、公共接口暴露方式。给出跨语言的目录结构骨架。

## 2. 数据流与状态管理

### 关注点
- 数据从哪里来？（用户输入、API、数据库、事件流）
- 数据如何在组件/模块间传递？（参数传递、全局状态、事件总线、消息队列）
- 状态在哪里维护？如何更新？（单向数据流、双向绑定、事件驱动）
- 数据转换发生在哪里？（序列化、验证、映射）

### 常见模式
- **单向数据流**：数据从顶层向下流动，事件向上冒泡
- **CQRS**：读写分离，命令和查询走不同路径
- **事件溯源**：状态变更以事件序列存储
- **响应式流**：数据作为可观察的流处理
- **仓库模式**：通过统一的仓库接口抽象数据访问

### 识别信号
- Store / Context / Provider 类文件 → 状态管理
- Event / Message / Queue 相关文件 → 事件驱动
- Repository / DAO / Mapper 文件 → 数据访问层
- DTO / Schema / Validation 文件 → 数据转换

### 输出要求
画出数据在系统中的流向图（文字描述），标注关键的状态存储点和转换节点。抽象为语言无关的数据流模式。

## 3. 错误处理策略

### 关注点
- 错误如何表示？（异常、Result 类型、错误码、错误对象）
- 错误如何传播？（抛出/捕获、返回值、回调、事件）
- 是否有统一的错误分类？（可恢复/不可恢复、业务错误/系统错误）
- 降级和容错策略？（重试、熔断、兜底值、优雅降级）
- 错误日志和监控如何集成？

### 常见模式
- **Result/Either 类型**：用类型系统强制处理错误
- **错误边界**：在特定层级捕获错误，防止扩散
- **全局错误处理器**：统一的错误拦截和转换
- **重试 + 退避**：对瞬态错误的自动恢复
- **错误分类链**：底层错误逐步转换为高层语义错误

### 识别信号
- `try/catch`、`Result<T>`、`Either<L,R>` → 错误表示方式
- `ErrorHandler`、`onError`、`catchError` → 错误处理点
- `RetryPolicy`、`CircuitBreaker` → 容错机制
- 自定义 Error 类继承 → 错误分类体系

### 输出要求
描述错误从产生到最终处理的完整链路，标注错误转换点。抽象为可跨语言应用的错误处理策略。

## 4. 可扩展性设计

### 关注点
- 如何在不修改已有代码的情况下添加新功能？（开闭原则）
- 是否有插件/扩展机制？如何注册和发现扩展？
- 中间件/拦截器模式的应用？
- 钩子（Hook）系统的设计？（生命周期钩子、事件钩子）
- 策略模式的使用？（可替换的算法/行为）

### 常见模式
- **插件注册表**：统一注册、发现、加载扩展点
- **中间件管道**：请求/消息经过一系列可组合的处理步骤
- **事件钩子**：在特定生命周期点触发用户自定义逻辑
- **策略注入**：通过配置或参数切换行为实现
- **适配器模式**：通过统一接口适配不同实现

### 识别信号
- `plugin`、`extension`、`addon` 目录 → 扩展机制
- `middleware`、`interceptor`、`handler` 文件 → 管道模式
- `register`、`subscribe`、`hook` 函数 → 注册机制
- 接口/抽象类 + 多实现 → 策略模式

### 输出要求
描述扩展点的设计、注册机制、扩展的生命周期。给出可跨语言实现的扩展框架骨架。

## 5. 配置管理

### 关注点
- 配置来源有哪些？（文件、环境变量、命令行参数、远程配置）
- 配置如何分层和覆盖？（默认值 < 配置文件 < 环境变量 < 命令行）
- 配置值的类型安全如何保证？（强类型配置对象 vs 松散键值对）
- 敏感配置如何处理？（密钥管理、加密、环境隔离）
- 配置变更是否支持热更新？

### 常见模式
- **配置层级叠加**：多层配置按优先级覆盖
- **强类型配置模式**：将配置映射到类型安全的结构
- **配置验证**：启动时校验配置完整性
- **环境隔离**：不同环境使用不同配置源
- **配置即代码**：配置文件纳入版本控制，支持审查

### 识别信号
- `.env`、`config.yaml`、`settings.json` → 配置文件
- `Config`、`Settings`、`Options` 类 → 配置对象
- `validate`、`parse`、`schema` 函数 → 配置验证
- `getEnv`、`fromEnv`、`env` 引用 → 环境变量集成

### 输出要求
描述配置的来源、层级、验证、类型安全机制。抽象为可跨语言实现的配置管理方案。

## 6. 测试策略

### 关注点
- 测试如何分层？（单元测试、集成测试、端到端测试的比例）
- Mock/Stub 策略？（哪些边界被 mock，哪些使用真实实现）
- 测试数据的准备方式？（工厂、固件、构建器）
- 测试命名和组织约定？
- 测试基础设施？（共享工具、自定义断言、测试运行器配置）

### 常见模式
- **测试金字塔**：大量单元测试、适量集成测试、少量 E2E
- **测试矩阵**：不同维度的组合测试
- **快照测试**：捕获输出结构，后续对比变更
- **Golden File 测试**：与参考文件对比
- **测试夹具工厂**：标准化测试数据创建

### 识别信号
- `__tests__`、`*.test.*`、`*.spec.*` 文件 → 测试文件位置
- `mock`、`stub`、`fake`、`fixture` → 测试替身
- `testUtils`、`testHelper`、`testing` 目录 → 测试基础设施
- `describe`、`it`、`test` 块 → 测试组织方式

### 输出要求
描述测试分层策略、Mock 边界、测试约定。抽象为可跨项目应用的测试策略。

## 7. API 与接口设计

### 关注点
- 公共 API 如何定义和暴露？（导出、接口定义、API 端点）
- 接口的稳定性如何保证？（版本化、向后兼容策略）
- 输入验证在哪里发生？（入口处统一验证 vs 内部分散验证）
- API 文档如何维护？（注释、类型定义、独立文档）
- 接口的粒度？（粗粒度 vs 细粒度，便利方法 vs 正交操作）

### 常见模式
- **门面模式**：简单的高层接口封装复杂的子系统
- **构建器模式**：通过链式调用逐步配置复杂对象
- **最小公开接口**：只暴露必要的操作，隐藏实现细节
- **版本化契约**：通过版本号或命名空间管理 API 演进
- **请求/响应对象**：用专用对象而非裸参数传递数据

### 识别信号
- `export`、`public`、`api` 目录/文件 → 公共接口
- `interface`、`protocol`、`contract` 定义 → 接口契约
- `validate`、`schema`、`parser` → 输入验证
- `version`、`v1`、`v2` 目录/前缀 → 版本化
- JSDoc、docstring、OpenAPI → API 文档

### 输出要求
描述接口设计原则、暴露策略、版本化方案。抽象为可跨语言应用的 API 设计模式。
