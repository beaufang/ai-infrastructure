本模板定义了架构提取器输出的 Markdown 文档结构。与传统的"架构文档"不同，这个模板以**可复现**为核心目标——读者应该能按照它在新项目中从零构建出相同的架构。

## 模板使用说明

### 必须遵守的规则

1. **构建顺序就是阅读顺序** — 模板按"地基→骨架→接线→扩展"组织，这就是实施顺序
2. **伪代码必须详细** — 不是三行注释式的伪代码，而是可以直接翻译为任何语言的详细逻辑
3. **核心类型必须有完整定义** — 字段、方法、类型关系必须清晰，不能只说"一个工具接口"
4. **接线必须有完整伪代码** — 展示具体的组装过程，不能只说"用依赖注入"
5. **请求路径必须可追踪** — 从入口到出口的每一步都要标注哪个组件在处理

### 关于深度

- **核心类型定义**：深度 = 可以直接翻译为代码。每个字段都有类型和用途说明。
- **核心组件实现**：深度 = 核心逻辑完整，边界条件处理清楚。不重要的细节可以省略。
- **外围功能**：深度 = 关键要点 + 与核心的连接方式。不需要完整实现。
- **反模式**：深度 = 具体代码表现 + 具体失败后果 + 正确做法。

### 关于"语言无关"

不是不能用具体语法——而是要解释清楚每个语法结构背后的意图，让不熟悉该语言的人也能理解：
- 用 `interface` 定义契约 → 意图是"定义组件间必须遵守的协议"
- 用泛型 `<T>` → 意图是"这个组件可以处理多种类型，具体类型由使用者决定"
- 用 `async/await` → 意图是"这个操作是异步的，调用者需要等待结果"
